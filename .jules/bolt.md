## 2024-05-18 - Optimize bulk IndexedDB fetching during hydrateStaticMetadataFn
**Learning:** During application initialization, `hydrateStaticMetadataFn` in `useLibraryStore` was issuing `Promise.all` over dozens or hundreds of individual `getBookMetadata(id)` calls. Because `getBookMetadata` internally opens its own `readonly` transaction, this created an "Asynchronous Cascade" that choked the microtask queue and severely degraded startup performance by spinning up N separate IDB transactions. Furthermore, a naive replacement using `.getAll()` would have introduced severe GC thrashing and OOM risks by loading the entire Blob-heavy database into main memory just to filter out the requested IDs.
**Action:** Created `getBookMetadataBulk` in `DBService` that opens a *single* transaction and executes parallel `.get(id)` and `.getKey(id)` requests via `Promise.all` inside the transaction boundary. Crucially, the returned array explicitly preserves the 1:1 index alignment with the input `ids` array, resolving both the performance bottleneck (dropping 1000 records from ~197ms to ~78ms) and maintaining API compatibility.
