# Versicle Overhaul Proposal — Contract-First

**Lens:** Harden the boundaries, then the internals. Define and freeze the system's hard contracts — persistence schema + versioned migrations, the CRDT document schema, worker RPC protocols, the TTS provider plugin interface, the store/selector public API, and an EPUB-rendering adapter that isolates epub.js — validate every contract at runtime (zod), version every contract, and then refactor internals freely behind stable, machine-enforced seams.

**Why this lens fits this codebase.** Versicle was built by AI agents over hundreds of iterations. The analysis shows the failure mode precisely: the *good* boundaries that exist (`src/lib/tts/engine/EngineContext.ts`, `replicationSpec.ts`, the three-domain IDB taxonomy, the `PlaybackBackend` seam) are exactly where quality is highest, and everywhere a boundary is missing — 97 `getState()` calls inside `lib/`, 65 madge cycles, `types/db.ts` importing the TTS engine, the dead zod validators, the five `useLibraryStore` race-regression files — is where the one-off-bug-fix sediment accumulated. Agents (and humans) write good code against contracts they cannot violate without a red CI. The overhaul therefore spends its budget making contracts *explicit, validated, versioned, and lint/CI-enforced first*, so that every subsequent internal rewrite — and every future agent iteration — is fenced.

Three non-negotiable properties, in priority order:

1. **Data safety** — user libraries (Yjs doc, IndexedDB binaries, backups, Firestore workspaces) survive every migration; every destructive path validates before it destroys; every dangerous flow is checkpointed.
2. **Hard-to-break guarantees** — every contract has a runtime validator, a version, and a contract test suite that both/all implementations must pass; boundary violations are build failures, not review comments.
3. **Replaceable parts** — epub.js, the TTS providers, the sync transport, the GenAI client, and the worker transport are each swappable behind one module without touching consumers.

### Boundary rules (machine-enforced, CI-blocking)

1. `types/` imports nothing. The `types/db.ts → lib/tts` inversion (root of ~50 of 65 madge cycles) is gone.
2. No module below L4 imports `store/` except L3 itself. `getState()` outside `store/` + `app/` is a lint error (today: 97 sites in `lib/`, 160 total outside `src/store`).
3. The worker runtime closure excludes `zustand`/`yjs`/`store/**` — asserted by a chunk-content test, not by `import type` discipline alone (which is currently the only guard, `EngineContext.ts:26-31`).
4. `idb`, `src/data/connection`, `src/data/schema` are unimportable outside `src/data/`; `'readwrite'` transactions are unwritable outside `write-gate.ts` (today raw `getDB()` lives in 7 modules including `GlobalSettingsDialog.tsx:246` and `TTSFlightRecorder.ts:165-227`).
5. `epubjs` is unimportable outside `src/reader/engine/` (today: 8+ files cast into rendition internals; `useEpubReader.ts` carries 40 eslint-disables).
6. Raw `fetch`/XHR is unimportable outside `src/lib/net/` (today: 11 fetch sites across 8 files plus an unbundled XHR loader in `public/piper/piper_worker.js`).
7. No module-scope side effects outside `app/bootstrap.ts` and entry files (today: importing any synced store boots Yjs persistence, `yjs-provider.ts:17,28-46`; importing `main.tsx` executes 229 of 266 modules).
8. `ui/` primitives import only Radix + `lib/utils` (today `ui/CompassPill.tsx:2-9` imports five domain stores and the Chinese dictionary).
9. One canonical import path per module (the `useLibraryStore.ts:798` re-export of `useBookStore` is deleted); path aliases everywhere.
10. No new test file outside the single convention (co-located `Foo.test.tsx`); one vitest config (`vitest.config.ts` merging vite config) — the worktree-exclusion fix currently lives in the dead config.

## How each confirmed debt is paid off

Debts are referenced by their analysis titles. Overstated findings are paid off at their recalibrated severity — they still get fixed, but they don't drive sequencing.

### Theme 1 — Data safety: the persistence contract (C1)

- **"'Clear All Data' leaves all user data behind (Yjs IDB never cleared)"** (critical) — `src/data/wipe.ts` `wipeAllData()`: `closeDB()` → `deleteDatabase('EpubLibraryDB')` + `yjsPersistence.clearData()`/`deleteDatabase('versicle-yjs')` + `localStorage.clear()` + Cache Storage purge. `GlobalSettingsDialog.tsx:240-262` stops enumerating store names; both reset entry points call the same function (also pays "**Clear-all-data duplicates DB schema knowledge in UI**" from app-shell). Ships in Phase 0 as a standalone hotfix — it is a broken privacy promise today.
- **"Backup restore wipes local data before validating replacement"** (critical) + **"Backup restore: unvalidated input, raw y-idb internals, magic sleeps, destructive-before-validated"** (high) — `src/data/snapshot/YjsSnapshotService.ts` is the single capture/validate/apply implementation: zod-validate the manifest, dry-run `Y.applyUpdate` on a scratch doc, auto pre-restore checkpoint via the existing `CheckpointService` flow, then apply; the y-idb fork gains explicit `flush()`/`whenSynced` (replacing both the raw `indexedDB.open('versicle-yjs')` schema re-implementation at `BackupService.ts:235-263` and the 1000 ms sleep at `:344`). `BackupService`, `CheckpointService`, and `android-backup` become thin format adapters — also paying **"Three parallel Yjs-snapshot capture/restore mechanisms"**.
- **"Backup restore corrupts cover images (ArrayBuffer JSON-stringified to {})"** (high) — backup manifest v3 strips/base64s binary fields in `generateManifest`; restore merges instead of blind-`put` (never overwrite a present local `coverBlob` with a non-binary value); a v2 reader is kept forever with `{}`-sanitization; a one-time boot repair nulls `{}` coverBlobs left by past restores so covers regenerate. Phase 0.
- **"WebKit-deadlock write lock is per-context and per-callsite, not an invariant"** (high) — `src/data/write-gate.ts` replaces the promise chain (`idb-write-lock.ts:24`) with `navigator.locks` (spans workers and tabs), exposed as `write(stores, syncCallback)` whose synchronous-callback signature *structurally* bans the intra-transaction awaits that hang WebKit (`DBService.ts:430-437` documents the probe-verified triggers). All bypassing writers (`ingestBook`, `deleteBook`, `saveLocations`, `cacheSegment`, `MaintenanceService.pruneOrphans`, `BackupService` restore writes, `CheckpointService`, `TTSFlightRecorder`) migrate through it. This is the precondition for the worker engine remaining the default topology safely. The detached-persistence behavior and its documentation are preserved verbatim (explicitly a keeper).
- **"Cached cloud-TTS alignment silently lost (alignmentData vs alignment drift)"** (recalibrated medium) — `CachedSegment` deleted; `getCachedSegment` returns `CacheAudioBlob` with one canonical field; cache write→read round-trip test asserts alignment survives. Latent, but a one-line class of fix — Phase 0.
- **"Schema strategy: idempotent-only upgrade destroys straggler user data"** (medium) + **"Unbounded caches: LRU fields exist but eviction never implemented"** (high) — IDB v25 introduces the versioned migration registry, snapshots legacy user-data stores before deletion, adds `blocked`/`blocking` handlers and `dbPromise` reset-on-failure; `navigator.storage.persist()` requested at first import; `audioCache` repo gains a size-budgeted idle-time LRU job over the already-maintained `lastAccessed` (run through the write gate); orphan scan extends to all per-book stores; `storage.estimate()` surfaced in settings.
- **"types/db.ts is a god module with dead types and inverted dependencies"** (high) + **"BookMetadata is a legacy intersection type with everything optional"** (high) — `types/db.ts` (934 lines, fan-in 59, imports `TTSQueueItem` *from* `AudioPlayerService`) dissolves into `src/data/rows/*` (zod-derived persisted shapes) and `types/user.ts`/`types/tts.ts`; the persisted queue-item shape lives in cache row types and `AudioPlayerService` imports it (arrow reversed). `BookMetadata` (intersection of three deleted-store row types) is replaced by `LibraryBook{availability: 'local'|'offloaded'|'ghost', …required display fields}` produced solely by `BookRepository` — also paying **"Central library selector returns any[]"** and **"Type-safety erosion: useAllBooks returns any[]"**.
- **"No data-access boundary: three competing tiers, raw getDB() in 7 modules"** (high) — the `src/data/` gateway plus boundary rule #4. **"Session-state mirror correctness gaps and dead code in DBService"** — `playbackCache` becomes the single session-state owner; seed-before-write always; flush-on-teardown via `visibilitychange`/web locks instead of the deliberate drop; the worker engine reaches it through an `EngineContext` persistence port (closing the documented exemption that currently breaks the cross-context lock).
- **"Three validation layers: one hand-rolled, one dead-and-drifted zod, none at trust boundaries"** (medium) — `src/data/rows/` is the single zod source of truth; the same schemas validate backup restore, Android payloads, and Firestore inbound; `sync/validators.ts` (dead, drifted — rejects `audio-bookmark` annotations) is rewritten against current shapes and *wired in*, or deleted with its 318-line fuzz suite if C2 validation supersedes it.
- **"Export utilities triplicated; two broken on native Android"** (medium) — `export-notes.ts` and `TTSAbbreviationSettings.tsx`'s hand-rolled `<a download>` both call the platform-aware `exportFile()` (a keeper).

### Theme 2 — The CRDT document schema contract (C2)

- **"Inbound hydration deletes state keys absent from the Y.Map — new fields can't be added safely"** (critical) — fork surgery: inbound patch merges map JSON over declared defaults instead of replace-with-delete (`yjs.mjs:187-209, 624-627`). This single change retires the entire class of `state.books || {}` defensive fallbacks (~20 sites), the migration-as-backfill hack (v4→v5 exists *only* to re-add `fontProfiles`), and the runtime lie in every synced store's TypeScript contract.
- **"Ephemeral popover state synced through CRDT to other devices"** (critical) — `syncedKeys` whitelist in the fork (declared per store in `defineSyncedStore`); popover moves to `useReaderUIStore`; v6 migration deletes the `popover` key from the `annotations` map.
- **"Migration runner is temporally fragile and fails silently"** (high) + **"Schema migration runner races its own version bumps and swallows failures"** (high) + layering's **"Yjs schema migrations behind nested dynamic imports"** — `src/app/migrations.ts`: static imports (legal at L4), strictly sequential `await`, one version bump inside one `yDoc.transact`, loud failure into safe mode, run exactly once by the coordinator after `whenHydrated()` (a new fork API) and before React renders. The double-`queueMicrotask` hack (`yjs-provider.ts:182-184`) and its spy-count test are deleted. A pre-migration checkpoint is taken automatically via the existing `CheckpointService` flow.
- **"Schema quarantine only guards the 'library' map with a corruption window"** (high) + sync's **"Obsolete-client quarantine covers 1 of ~10 synced maps and never disconnects the provider"** (high) — schema version moves to a dedicated `meta` Y.Map checked **synchronously in `FirestoreSyncManager` before applying any remote update**; on obsolete, the provider is actually detached and the device heartbeat stopped (today `handleObsoleteClient` only flips a status label); workspace metadata `schemaVersion` updates post-migration; the UI lock becomes a direct synchronous import. Dual-write `library.__schemaVersion` for one release so v5 clients still quarantine.
- **"Write amplification: full-state deep diff per set() against growing progress map"** (medium) — per-key scoped diffing in the fork; `readingSessions` history moves out of the hot progress record (or is capped far lower). This is also what lets the view-model derivation (Theme 6) stop being heroic.
- **"Per-device preferences create permanent undeletable top-level Y.Maps"** (medium) — v6 folds `preferences/<deviceId>` maps into one `preferences` map keyed internally by deviceId; old maps become empty husks (Yjs can't delete top-level types) but stop growing; `deleteDevice` can finally delete a device's data.
- **App-shell's "Per-device UI state (activeContext) synced via Yjs"** (recalibrated medium — it is per-device-keyed, not cross-device-flipping) — `activeContext` leaves the CRDT anyway as part of making Notes routable (`/notes`); per-device navigation state has no business in a synced doc.

### Theme 3 — Sync transport, server rules, recovery (C3)

- **"firestore.rules catch-all neuters tombstone protection; rule syntax invalid"** (critical) — rules rewritten with non-overlapping matches and correct syntax (`resource == null`, `'isDeleted' in resource.data`), covering `updates`/`history`/`maintenance`/`metadata`; validated by a Firestore-emulator rules suite in CI. Today the recursive wildcard at `firestore.rules:35-37` makes every deny unreachable and the tombstone invariant is client-side fiction.
- **"No Cloud Storage rules/deploy story though snapshots live in Cloud Storage; deleteWorkspace leaves data behind"** (critical) — ship `storage.rules` (owner-only) + `firebase.json` + documented deploy; `deleteWorkspace` purges `history`/`maintenance`/`metadata` and Cloud Storage `snapshot_vN.bin` blobs; a one-time "purge deleted workspaces" maintenance action cleans residue from past deletions. Emulator integration test for deletion.
- **"Workspace switch has a data-loss window and a rollback that can silently fail"** (critical) — staged-swap protocol: download → staging IDB → verify decode → atomic swap → reload; the migration's checkpoint gains a `protected: true` flag excluded from the rolling-10 prune; `navigator.locks` around restore; migration state persists until restore *succeeds*. The migration state machine concept (localStorage bridging the reload, confirm/rollback UX, ErrorBoundary → `CriticalMigrationFailureView`) is a keeper — only its holes are patched.
- **"deleteWorkspace kills the whole manager and severs the active workspace even when deleting another one"** (high) — conditional sever; `disconnectFireProvider()` scoped to the deleted path; reconnect after delete. Ships in Phase 0 (no data-shape change).
- **"FirestoreSyncManager is a 993-line god object with duplicated temp-doc hydration and unawaited throws"** (high) — decomposed into `AuthSession` / `WorkspaceService` / `ProviderConnection` over `SyncBackend`, with one shared `downloadWorkspaceState(path)` temp-doc utility; all async paths awaited.
- **"Mock test mode interleaved through production code; mock drifted from real provider"** (high) — `SyncBackend` injection at the composition root deletes all 8 inline `__VERSICLE_MOCK_FIRESTORE__` branches; `MockFireProvider` leaves the production import graph (it is statically imported by `FirestoreSyncManager.ts:31` today and ships in the user bundle); both backends pass `describeSyncBackendContract`.
- **"Forked sync stack pinned to moving branch refs"** (recalibrated medium — lockfile pins SHAs and CI uses `npm ci`, but branch refs silently jump on lockfile refresh) — the three forks are vendored as npm workspaces (`packages/y-cinder`, `packages/y-idb`, `packages/zustand-middleware-yjs`) with upstream LICENSE files retained verbatim and fork-provenance headers (the licensing workstream gates this PR); `yjs`/`zustand` become peer deps with a single-yjs-instance CI assertion; the emulator contract suite is the forks' acceptance gate. Vendoring is also what makes the C2 fork surgery a first-party, tested change instead of a remote branch bump.
- **"Fragmented initialization and duplicated status plumbing"** (medium) — `SyncOrchestrator.boot()` is the single lifecycle owner registered as a C11 boot task (today `initialize()` is called from three places with different gating, and `App.tsx:198` ignores the enabled flag); `lastSyncTime` driven by provider flush events; the manager stops importing `useToastStore` (8 sites) — it emits typed `SyncEvent`s the UI translates.
- **"useSyncToasts serializes the entire progress map on every store update"** (medium) — selector-based subscription with per-book shallow compare (or origin-tagged remote-change events); the aliased `useLibraryStore` import path normalized.
- **"SyncSettingsTab is a 661-line god component"** (medium) — split into Device/Connection/Workspace sections; the embedded Drive picker/scanner and Google client-ID config move to the google subsystem's settings panel; the duplicate `FirebaseConfig` interface deleted.
- **"Circular imports and stale persistence binding in the checkpoint path"** (medium) — `CheckpointService` accepts an injected `pauseSync()` handle instead of importing the manager; `getYjsPersistence()` accessor replaces the const snapshot.
- **type-safety's "Zod sync validators are dead code; remote payloads trusted blindly"** (recalibrated high) — paid by C2/C3 validation: `WorkspaceMetadata` validated on read; post-merge Yjs entity validation with quarantine + flight-recorder snapshot, shipped observe-then-enforce so real-world data variance is measured before anything is rejected.

### Theme 4 — Worker RPC + TTS engine contract (C4)

- **"Dual-transport contract unowned; parity suite covers a fraction of behavior"** (high) — `TtsEngine` becomes a standalone interface implemented by `WorkerEngineHandle` (production) and the in-process service (tests only — `mainThreadAudioPlayer.ts:29-34` already forces the worker in production). Commands are acks; handle errors surface in the snapshot error channel. `engineParityScenarios.ts` grows into the contract spine and absorbs the per-bug regression files.
- **"Two notification paths emit inconsistent snapshots"** (high) — one immutable `PlaybackSnapshot` with a monotonic `seq`, emitted from exactly one place; errors and download progress become fields, not parallel positional callbacks (today `setStatus`, the PSM subscription, `notifyError`, and `notifyDownloadProgress` each build their own tuples — the "treat loading as playing" flicker workaround in `useTTSStore.ts:242-245` is a symptom).
- **"Replication echo loop: full store (incl. queue) re-serialized into the worker per sentence"** (high) + state-stores' **"useTTSStore conflates persisted settings, live engine mirror, and command facade"** (high) — `useTTSStore` splits into `useTTSSettingsStore` (persisted; replicated as the explicit data-only `TTSSettingsData` payload) and `useTTSPlaybackStore` (engine mirror; never replicated). ## Migration roadmap

## Test strategy for the end state

The test suite converges on the **engine-parity template** (`engineParityScenarios.ts` + typed fakes + DI seams) — the one pattern in this codebase that already works — generalized into a six-tier pyramid. Headline: every frozen contract has exactly one suite that *is* its specification; regression knowledge moves from 246 scattered files into tagged describes inside those suites (~110 files).

**Tier 1 — Pure-logic units (vitest, node env).** One file per module + `.fuzz` companions on the seeded infrastructure (`src/test/fuzz-utils.ts`, a keeper): CFI algebra (fast-path vs parsed-reference equivalence — the gate for every string optimization), `progress/resolve.ts`, lexicon application, text segmentation/refinement, K-means palette, json-diff. Benchmarks become `*.bench.ts` outside the gate.

**Tier 2 — Contract suites (the spine).** `describeXContract(makeHarness)` for every dual-implementation boundary:
- `TtsEngine`: parity scenarios over in-process + real Comlink/MessageChannel worker (restore, skip masks, adaptations, navigation, dragnet, fallback, snapshot monotonicity, flight export). `vi.mock` banned in the engine directory; fakes only.
- `ITTSProvider`: all five providers pass one suite (play/preload semantics, single-shot failure, dispose, speed policy, cache-key shape); the Capacitor smart-handoff and WebSpeech voiceschanged behaviors (keepers) fold in as named scenarios.
- `SyncBackend`: `FirestoreBackend` against the Firebase emulator (nightly + on fork bumps: connect, compact, tombstone-denial, delete-purge, rules) and `MockBackend` per-PR — both on the same spec, which also serves as the vendored forks' acceptance gate.
- `zustand-middleware-yjs` bridge: hydration merge, syncedKeys, per-key diff, echo prevention, `whenHydrated`, obsolete quarantine — in the fork workspace, run by this repo's CI.
- Storage repos against fake-indexeddb + the real-browser `_idb_probe.js` WebKit lane; `ReaderEngine` conformance run on `EpubJsEngine` (jsdom fixtures) and `FakeReaderEngine`.
- Replication pinning stays as-is: `replicationSpec` + `replication.test.ts` fail loudly on one-sided additions (keeper).

**Tier 3 — Service/store integration.** Real stores + real Y.Doc + typed harness doubles from `src/test/harness/` (replacing the 712 inline `vi.mock` blocks; inline mocking of DBService/stores lint-banned): `LibraryService.concurrency.test.ts` (the lifted race invariants), `ProgressService` projections, migration coordinator against doc-snapshot fixtures, `NetworkGateway` consent/timeout behavior, GenAI feature modules against `MockGenAIClient` with zod-validated round-trips.

**Tier 4 — Component tests** only where real logic lives, via one `renderWithStores()` helper with real Zustand stores; vitest-axe smoke baked into the shared harness; presentational tabs (the `TTSSettingsTab` model) need no mocks by construction.

**Tier 5 — ~40 Playwright journeys.** Deterministic waits through the single `VITE_E2E`-gated `window.__versicleTest` API (`flushPersistence`/`resetApp`/`seedLibrary`) — no sleeps encoding other subsystems' debounce intervals. Sanitization ON; one hostile-EPUB journey (XSS + tracking-pixel fixture); desktop per PR, mobile+WebKit+Firestore-emulator sync journey nightly; data-testid discipline preserved (keeper).

**Tier 6 — ~10 `toHaveScreenshot` goldens** (library, reader, themes, settings, audio deck); all other captures failure-only.

**CI shape:** lint + `tsc -b` (app, tests, e2e, configs) + sharded vitest + Playwright desktop on every PR with `npm ci`, pinned Node, `CI=1` propagated; nightly emulator + mobile lanes + scheduled Docker image builds; ratchets on coverage, production `as any` (138 → 0 outside the epubjs adapter), eslint-disables (245 → ~0), entry-chunk size, and vitest file count; the worker-chunk purity test and the registry==CSP test run per PR. Every contract's version bump requires a matching contract-suite change in the same PR — that rule, plus the boundary lint, is what makes the system hard to break for the agents that will keep building it.
